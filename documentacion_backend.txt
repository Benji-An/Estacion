1. ARQUITECTURA Y FLUJO DE COMUNICACIÓN
-----------------------------------------------------
El backend sigue un patrón de arquitectura limpia dividida en capas. Aquí te explicamos cómo se comunica cada archivo:

1. app/main.py (Punto de entrada): 
   Levanta la aplicación, configura CORS para que el frontend pueda conectarse y agrupa todos los enrutadores (endpoints).
   
2. app/api/v1/endpoints/ (Controladores):
   Son las "puertas" del backend. Reciben los requests (peticiones) del frontend, validan rápidamente los parámetros de entrada y deciden a qué servicio o función CRUD llamar. No deben contener lógica matemática pesada.
   
3. app/schemas/ (Esquemas Pydantic):
   Sirven como validadores de información entre el frontend y el backend. Definen con exactitud qué campos debe enviar el cliente (ej. un email y un password) y qué se le responderá.

4. app/services/ (Lógica de Negocio):
   Aquí vive la inteligencia de la aplicación.
   - csv_service.py: Se encarga exclusivamente de leer los archivos CSV de las estaciones HOBO, decodificarlos, buscar en los encabezados complejos (mediante etiquetas "LBL:") y convertirlos en listas de objetos limpios en Python.
   - hydro_service.py: Ejecuta la fórmula del Método Racional (Q = C × I × A / 360) para obtener los caudales de escorrentía, y expone la tabla de coeficientes del IDEAM.
   - idf_service.py: Contiene toda la estadística pesada. Lee series de tiempo, calcula la intensidad de la lluvia por ventanas (5 a 120 min) y genera las Curvas de Intensidad-Duración-Frecuencia (IDF) aplicando la distribución de Gumbel para distintos periodos de retorno.

5. app/crud/ (Acceso a Datos):
   Son los intermediarios entre los Servicios/Endpoints y la Base de Datos PostgreSQL.
   - archivo.py: Guarda registros cuando se sube un nuevo CSV.
   - medicion.py: Inserta de a miles (bulk insert) los datos leídos por csv_service. También calcula resúmenes estadísticos (agregaciones de SQL como AVG, SUM, MAX) y saca series temporales para las gráficas.

6. app/models/ (Modelos SQLAlchemy):
   Representan las tablas de la base de datos (Usuario, ArchivoCSV, MedicionEstacion).

FLUJO DE EJEMPLO (Subir un CSV y ver curvas IDF):
Frontend envía CSV -> `datos.py/upload` -> llama a `csv_service.py/parse_estacion_hobo` -> llama a `crud/medicion.py/create_bulk` -> El archivo queda guardado en PostgreSQL.
Luego Frontend pide las curvas -> `calculo.py/idf` -> lee datos de `crud/medicion.py` -> los envía a `idf_service.py/calcular_curvas_idf` -> Retorna un JSON listo para que Recharts o Chart.js lo grafiquen.


2. LISTA DE ENDPOINTS DISPONIBLES (PARA CONSUMIR EN FRONTEND)
-----------------------------------------------------
Todas las rutas tienen el prefijo base: /api/v1

AUTENTICACIÓN
[POST] /login/access-token
- Descripción: Endpoint para iniciar sesión.
- Body (Form Data): `username` (email del usuario) y `password`.
- Retorno: `{"access_token": "eyJhbG...", "token_type": "bearer"}`

[POST] /register
- Descripción: Crear un nuevo usuario en la base de datos.
- Body (JSON): `{"email": "correo@ejemplo.com", "password": "mipassword", "full_name": "Nombre"}`
- Retorno: Información del usuario creado (sin contraseña).


GESTIÓN DE DATOS Y ESTACIÓN HOBO
[POST] /datos/upload
- Descripción: Subir y procesar un archivo CSV.
- Autenticación: Requiere Header -> `Authorization: Bearer <token>`
- Body: `multipart/form-data` con un campo llamado `file`.
- Retorno: Metadata del archivo procesado (id, nombre, total_registros).

[GET] /datos/archivos
- Descripción: Listado paginado de los archivos CSV ya subidos.
- Parámetros URL (Opcionales): `?skip=0&limit=20`
- Retorno: Lista de archivos. Público.

[GET] /datos/mediciones
- Descripción: Obtener datos en bruto de un CSV específico.
- Parámetros URL (Obligatorio): `?archivo_id=1`
- Parámetros URL (Opcionales): `?skip=0&limit=100`
- Retorno: Lista de mediciones paginadas. Público.

[GET] /datos/resumen
- Descripción: Devuelve métricas generales de un CSV (lluvia total, temp media, etc).
- Parámetros URL (Obligatorio): `?archivo_id=1`
- Retorno: Objeto JSON con KPIs. Público.

[GET] /datos/series
- Descripción: Endpoint diseñado para gráficas en el frontend.
- Parámetros URL (Obligatorio): `?archivo_id=1`
- Parámetros URL (Opcional): `?campo=lluvia_mm` (Opciones: contenido_agua, rad_solar, temp_c, humedad_pct, vel_viento, vel_rafagas, dir_viento, lluvia_mm).
- Retorno: Arreglo de puntos `[{ "timestamp": "...", "valor": 12.5 }, ...]`. Público.


CÁLCULOS HIDROLÓGICOS (MÉTODO RACIONAL & IDF)
[POST] /calculo/escorrentia
- Descripción: Calcula el caudal de escorrentía (Q = C*I*A/360).
- Body (JSON): `{"C": 0.60, "I": 50.5, "A": 10.0}`
- Retorno: Objeto JSON con el caudal Q en m³/s y L/s. Público.

[GET] /calculo/coeficientes
- Descripción: Lista de los coeficientes "C" de escorrentía típicos según IDEAM.
- Retorno: Objeto con opciones (ideal para mostrar un Dropdown/Select en el UI). Público.

[GET] /calculo/intensidades
- Descripción: Calcula las intensidades máximas de lluvia para la duración de 5, 10, 15, 30, 60 y 120 minutos basándose en un archivo subido.
- Parámetros URL (Obligatorio): `?archivo_id=1`
- Retorno: Array con las intensidades observadas en mm/h. Público.

[GET] /calculo/idf
- Descripción: Genera las Curvas IDF (Intensidad-Duración-Frecuencia) de un CSV aplicando distribución Gumbel para periodos de retorno de 2 a 50 años.
- Parámetros URL (Obligatorio): `?archivo_id=1`
- Retorno: Un objeto con la estructura óptima para el Frontend:
  - `tabla`: Lista por duraciones, útil para armar una tabla de datos HTML.
  - `por_periodo`: Diccionario agrupado por Años (2, 5, 10...), ideal para pasar a Chart.js o Recharts directamente en las líneas de la gráfica.
- Público.
